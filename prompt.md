# Evaluation Prompt — HR-сервис управления персоналом

## ЗАДАЧА

Оценить выполненное практическое задание по разработке HR-сервиса управления
персоналом по шкале 0–100. Оценивается не только работающий сервис, но и весь
процесс: как студент декомпозировал задачу, как работал с ИИ-агентом, какие
решения принимал и как их документировал.

Порог прохождения:

- Ниже 65 — работа не соответствует базовым требованиям.
- 65–80 — базовый рабочий уровень.
- 80–90 — хороший результат.
- 90–100 — очень качественная реализация.

При критических ошибках или низком качестве решения работа должна быть
оценена не более чем в 60 баллов.

---

## ОБЩИЕ ПРАВИЛА ВЫВОДА

- Оценивать строго по критериям ниже, без отклонений.
- Применять автоматические штрафы до итоговой оценки.
- Не наказывать за отсутствие красивого дизайна клиентской части.
- Не наказывать за выбор «непопулярного» стека, если он обоснован.
- Не наказывать за простую авторизацию (логин/пароль в конфиге).
- Не требовать дословного сохранения промптов — достаточно краткого
  отражения подхода и результата в логе разработки.
- Оценку привязывать к содержанию файлов, а не к их конкретным именам.
  Структура должна быть понятной и следовать логике: спецификация отдельно,
  лог разработки отдельно, правила агента в корне, инфра в infra/.
- Инфраструктуру оценивать по корректности написанных файлов конфигурации,
  а не по факту запуска — возможности запустить Docker на машине
  проверяющего нет.
- Публичный адрес сервиса брать из docs/README.md — там должен быть раздел
  с провайдером, IP/доменом и описанием проксирования. Если адрес указан —
  проверить доступность сервиса по нему.
- Главный ориентир: «Видно ли по репозиторию, что человек осознанно управлял
  процессом разработки, а не просто нажал "сгенерировать"?»

## КРИТЕРИИ ОЦЕНКИ

### 1. Качество промптов и работа с агентом (0–30)

Оцени, насколько осознанно студент управлял ИИ-агентом.

Учитывай:

- в корне репозитория есть файл правил для агента (.cursorrules или аналог)
  с осмысленными инструкциями под задачу, а не пустой или шаблонный
- в логе разработки отражено, какой подход использовался на каждом этапе
  и что из этого вышло — принято, отклонено, доработано
- видно, что студент итерировал и управлял агентом, а не просто
  скопировал ТЗ в чат и принял первый ответ
- дословные промпты не обязательны, но рефлексия по каждому этапу должна
  присутствовать

Сильное снижение баллов:

- файл правил для агента отсутствует или содержит только шаблонный текст
- лог разработки не отражает работу с агентом — только список сделанного
- нет никакой рефлексии: что работало, что нет, что пришлось переделывать

### 2. Декомпозиция задачи и планирование (0–15)

Оцени, как студент разбил задачу на части до начала разработки.

Учитывай:

- в папке docs/ есть спецификация или ТЗ, составленное до первого коммита
  с кодом (видно по истории коммитов)
- спецификация содержит схему БД, описание эндпоинтов, описание экранов
- лог разработки ведётся отдельно от спецификации или структурно отделён
  внутри одного файла — планирование и реализация не смешаны в кучу
- история коммитов показывает пошаговое развитие, а не один большой дамп
- коммитов не менее 10, сообщения осмысленные и следуют формату из ТЗ

Сильное снижение баллов:

- спецификация отсутствует или добавлена одним коммитом в конце
- папка docs/ пустая или содержит только README
- коммиты — один-два огромных «initial commit» и «fix»
- нет видимой логики в порядке разработки

### 3. Качество реализации сервиса (0–30)

Оцени полноту и корректность реализации всех трёх компонентов системы.

#### Backend API

Учитывай:

- реализованы все 5 обязательных эндпоинтов:
  GET /employees, GET /employees/:id, POST /employees,
  PUT /employees/:id, GET /departments
- GET /employees поддерживает фильтрацию по отделу, статусу и руководителю
  одновременно, параметры передаются через query string
- GET /employees реализует пагинацию: параметры page/limit или offset/limit,
  в ответе возвращается общее количество записей
- GET /departments возвращает список отделов с количеством сотрудников
  в каждом, а не просто список названий
- POST /employees и PUT /employees/:id валидируют входные данные:
  обязательные поля, форматы дат, допустимые значения статуса
- seed-файл или миграция содержит все 12 записей из ТЗ с полными данными:
  ФИО, отдел, должность, дата найма, зарплата, статус, email, руководитель
- код API читаемый, логика не перемешана в одном файле

Сильное снижение баллов:

- часть обязательных эндпоинтов отсутствует
- фильтрация или пагинация не работают или реализованы некорректно
- seed-файл отсутствует или содержит неполные данные
- нет валидации входных данных

#### Клиентская часть

Учитывай:

- отображается список сотрудников, загружаемый с API
- работает фильтр по отделу: при выборе отдела список обновляется
- работает фильтр по статусу: при выборе статуса список обновляется
- фильтры можно применять одновременно
- пагинация работает: есть навигация по страницам или подгрузка
- карточка или строка сотрудника содержит основные поля:
  ФИО, отдел, должность, статус
- интерфейс удобен для использования: фильтры очевидны,
  состояние загрузки обозначено, пустой результат обработан

Сильное снижение баллов:

- список сотрудников не загружается или загружается без обращения к API
- фильтры присутствуют в интерфейсе, но не работают
- пагинация отсутствует

#### Административная панель

Учитывай:

- панель защищена авторизацией: без логина доступ закрыт
- присутствует форма добавления нового сотрудника со всеми полями из ТЗ
- присутствует форма редактирования существующего сотрудника,
  поля предзаполнены текущими данными
- формы отправляют данные в API (POST /employees и PUT /employees/:id)
- после успешного сохранения есть обратная связь пользователю

Сильное снижение баллов:

- авторизация отсутствует или обходится без пароля
- форма добавления или редактирования отсутствует
- формы не обращаются к API

#### Тесты

Учитывай:

- написаны минимум 3 автотеста на ключевые эндпоинты
- тесты покрывают базовые сценарии: получение списка, получение одной
  записи, создание записи
- тесты запускаются командой из документации или стандартной командой
  тест-фреймворка
- тесты включены в CI/CD пайплайн

Сильное снижение баллов:

- тесты отсутствуют
- тесты есть, но не запускаются или падают

### 4. Инфраструктура и доступность сервиса (0–15)

Оцени корректность конфигурационных файлов и доступность задеплоенного сервиса.

Учитывай:

- в docs/README.md есть раздел сервера: провайдер, публичный IP/домен,
  описание проксирования (/ → порт frontend, /api/ → порт backend,
  /admin/ → порт admin). Проверить доступность сервиса по указанному адресу.
- docker-compose.yml описывает все необходимые сервисы (backend, frontend,
  admin, база данных), порты и зависимости между ними корректны
- переменные окружения вынесены в .env, в репозитории лежит .env.example
  с описанием всех переменных
- infra/nginx.conf присутствует, маршруты описаны корректно и соответствуют
  портам в docker-compose.yml
- CI/CD файл в .github/workflows/ присутствует, шаги реалистичны:
  линтер, тесты, деплой при пуше в main
- seed-файл или миграция с тестовыми данными из ТЗ присутствует

Сильное снижение баллов:

- адрес сервиса не указан в README или сервис недоступен по указанному адресу
- docker-compose.yml отсутствует или содержит явные ошибки
  (неверные порты, отсутствующие зависимости, опечатки в именах сервисов)
- секреты захардкожены в коде или конфигах репозитория
- infra/nginx.conf отсутствует
- CI/CD отсутствует или содержит нереалистичные шаги

### 5. Рефлексия в документации (0–10)

Оцени, насколько осмысленно документирован процесс разработки.

Учитывай:

- папка docs/ структурирована: спецификация и лог разработки либо в разных
  файлах, либо чётко разделены внутри одного
- лог разработки содержит честные записи по этапам: что делали, какой подход
  использовали с агентом, что получилось, что пошло не так
- обоснование решений (стек, архитектура) написано своими словами
- если есть папка references/ — она содержит реальные материалы,
  а не пустая директория для галочки
- docs/README.md содержит быстрый старт, описание переменных окружения,
  краткое описание эндпоинтов и раздел про деплой и сервер
- документация выглядит как живой рабочий артефакт, а не отчёт постфактум

Сильное снижение баллов:

- папка docs/ пустая или содержит только README без спецификации и лога
- лог разработки добавлен одним коммитом в конце
- лог формальный: перечисление сделанного без рефлексии и описания процесса
